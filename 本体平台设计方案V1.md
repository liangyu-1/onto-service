

# 本体存储查询引擎 

![image.png](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/pLdn55XRRzpN5no8/img/0fc175b3-b68e-4cb7-b307-80501e355dbd.png)

本章节描述一套面向工业场景的本体存储与查询引擎方案。系统底层使用 Doris 存储和执行，语义层使用 Property Graph / Semantic SQL 进行 TBOX 建模、查询、推理和 Action/tool 编排。

核心定位如下：

*   Doris 负责 ABOX 物理存储、OLAP、物化视图、DML 和执行。
    
*   Ontology Service 负责  TBOX、ABOX的全部操作。
    
*   Semantic  layer 负责 SQL / Semantic SQL 的 parse、analyze、resolve 和 rewrite。
    
*   Logic Registry 负责管理规则定义、外部任务绑定。
    
*   Query Adapter  负责推理事实查询和解释模板。
    
*   Action Gateway 负责管理 LLM tool contract、外部动作绑定、dry-run 请求路由和调用记录。
    
*   外部大数据平台负责定时调度、复杂 Logic 执行、物化产出、动作编排和工作流执行；本体层只保存定义、绑定和可查询结果引用。

## Ontology Service

### Semantic  layer 

Semantic Layer 是语义控制面。

核心职责：

*   有哪些业务对象类型。
    
*   对象有哪些属性。
    
*   对象之间有哪些谓语关系。
    
*   哪些别名、业务术语、中文说法指向哪个语义对象。
    
*   TBOX 定义如何映射到 Doris ABOX 表、视图、列和关系。
    
*   LLM 应该如何理解每个对象、属性、关系和规则。
    

Semantic Layer 不直接承担 OLAP 执行。

主要维护和查询这些元数据：

```text
ontology_domain
ontology_node_type
ontology_property
ontology_edge_type
ontology_class_abox_mapping
ontology_ai_context
ontology_logic
ontology_action
```

对 LLM 来说，Semantic Layer 的价值是 grounding。

例如用户说：

```text
查 P1 过程里攻击窗口内 P1_B2004 的最近值
```

Semantic Layer 负责把它 grounding 成：

```text
P1 -> ICSProcess 实例
P1_B2004 -> SCADAPoint 实例
攻击窗口 -> AttackWindow
最近值 -> PointSample.value
```

然后再把语义对象交给 Query Adapter 生成可执行查询。

### Logic Registry 

Logic Registry 是规则控制面。

核心职责：

*   规则定义。
    
*   指标定义。
    
*   约束定义。
    
*   状态机定义。
    
*   告警规则定义。
    
*   Logic 依赖。
    
*   外部执行绑定。
    
*   结果表位置。
    
*   推理事实定义。
    
*   解释模板。
    

Logic Registry 解决的问题是：

```text
某个语义属性或推理事实是怎么算出来的。
它依赖哪些 DATA、其他 LOGIC 或外部任务。
它由谁执行。
结果写到哪里。
LLM 查询时应该如何解释。
```

例如：

```text
ICSProcess.security_state
```

在 Logic Registry 里是一条规则定义。

它可能包含：

```text
target_type = ICSProcess
target_property = security_state
implementation_type = sql / udf / python / nlp / external_service
dependencies = AttackWindow, PointSample.attack_label, SCADAPoint.anomaly_state
execution_binding = hai_security_state_job
result_table = semantic_fact_current
explanation_template = security_state_explanation_zh
```

### Query Adapter

Query Adapter 是语义查询到物理查询的适配层。

它负责 agent 的输入生成DORIS 可识别的 SQL：

```text
Semantic Query / Graph Query / Logic Query
  -> resolved semantic plan
  -> Doris SQL
  -> ABOX rows / facts / evidence
```

Query Adapter 是 TBOX 和 ABOX 的桥。

它查 TBOX 是为了知道“怎么查”。

它查 ABOX 是为了拿到“真实数据”。

### Action Gateway

Action Gateway 是动作控制面。

核心职责包括：

*   查询可用 tool。
    
*   描述 tool 的输入输出。
    
*   根据 TBOX ACTION 校验目标对象、输入 schema 和前置条件。
    
*   根据 ABOX DATA / Derived Facts 执行 dry-run 检查。
    
*   将用户确认后的动作提交给外部平台。
    
*   镜像外部动作状态。
    
*   写入 action instance 和 action log。
    

Action Gateway 必须强制区分：

```text
dry-run:
  只校验，不真正执行外部动作。

submit:
  用户确认后，把请求提交给外部大数据平台或工业系统。

audit:
  记录谁在什么时候基于什么证据触发了什么动作。
```

### TBOX 数据层

TBOX DATA 存业务对象定义。

核心对象包括：

*   Object Type。
    
*   Ontology Domain。
    
*   Property。
    
*   Relationship。
    
*   Object Type 到 ABOX object source 的映射。
    
*   Property 到 ABOX column / expression 的映射。
    
*   Relationship 到 FK / edge table / ROW / MULTIROW 的映射。
    
*   `ai_context`。
    

TBOX DATA 的主源包括：

```text
CREATE Ontology Domain
OBJECT_TYPE TABLES
RELATIONSHIP TABLES
PROPERTIES TABLES
MAPPINGS TABLES
OPTIONS(ai_context = ...)
```

也支持先以元数据表表达，再生成 GoogleSQL Catalog。

#### 图 / namespace 定义表

```sql
ontology_domain (
  domain_name, // 本体图名称，例如 PlantGraph
  version, // TBOX 版本号，例如 1.0.0
  ddl_sql, // 原始 Property Graph / Semantic SQL DDL
  status, // 版本状态：draft / validated / published / deprecated / archived
  created_at, // 创建时间
  created_by, // 创建人或发布服务账号
  ddl_hash // DDL 内容 hash，用于变更检测和幂等发布
);
```

用途：

保存 TBOX DATA 的源定义。

支持版本管理。

支持发布和回滚。

支持 LLM 判断当前应该使用哪个本体版本。

#### 节点类型表

```sql
ontology_object_type (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  label_name, // 节点类型 / class / label 名称，例如 Equipment
  parent_label, // 父类 label，用于 class 继承，例如 PhysicalAsset
  display_name, // 面向用户显示的类型名称
  description, // 面向人类的简短说明
  ai_context // 面向 LLM 的类型语义、别名、使用建议和反例
);
```

说明：

`ontology_object_type` 表达 class 本身。

ABOX 物理来源由独立映射表表达。

class 的 ABOX 承载方式放在 `ontology_class_abox_mapping`。

#### object 到 ABOX 映射表

```sql
ontology_object_abox_mapping (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  class_name, // TBOX class / label 名称，例如 BoilerProcess
  parent_class, // 父类名称，例如 ICSProcess / HAIComponent
  mapping_strategy, // 承载策略：single_wide_table / class_view / class_table / union_view
  object_source_name, // ABOX 对象源名称，可以是 Doris 物理表、逻辑视图或物化视图
  source_kind, // 对象源类型：physical_table / logical_view / materialized_view / table_function
  primary_key, // ABOX 对象源主键列，例如 process_id / point_tag
  discriminator_column, // 单宽表继承的类型判别列，例如 process_id / point_prefix
  type_filter_sql, // 子类实例过滤条件，例如 process_id = 'P1'
  property_projection_json, // class 可见属性到 ABOX 列/表达式的投影
  view_sql, // class_view / union_view 的定义 SQL
  materialization_strategy, // 虚拟表是否物化以及刷新策略，例如 virtual / materialized_5m
  ai_context // 面向 LLM 的说明：实例从哪里来、不要误用哪些列
);
```

该表是 DATA 层核心映射表，用于描述 class 到 ABOX source 的承载策略。

支持的策略：

```text
single_wide_table:
  父类和子类都来自一张最宽表。

class_view:
  每个 class 暴露为一个逻辑视图或物化视图。

class_table:
  每个 class 对应一张独立物理表。

union_view:
  父类通过 UNION ALL 汇总多个子类表。
```

#### 属性表

```sql
ontology_property (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  owner_label, // 属性所属类型 / label，例如 SCADAPoint
  property_name, // 语义属性名，例如 anomaly_state
  value_type, // 属性值类型，例如 STRING / DOUBLE / TIMESTAMP
  column_name, // 直接映射的 Doris 物理列；表达式属性可为空
  expression_sql, // 派生属性或 measure 的 GoogleSQL 表达式
  is_measure, // 是否为聚合指标 / measure
  semantic_role, // 语义角色：identifier / dimension / state / metric
  semantic_aliases, // 属性别名数组，辅助 LLM grounding
  hidden, // 是否默认隐藏，适用于敏感字段或内部字段
  description, // 面向人类的简短说明
  ai_context // 面向 LLM 的业务含义、使用建议和歧义消解
);
```

说明：

直接属性映射到 Doris 列。

派生属性来自 LOGIC。

measure 由 LOGIC 或 Doris 聚合计算。

子类属性映射到宽表中的子类专属列。

#### 关系表

```sql
ontology_relationship (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  label_name, // 关系 / 谓语 / edge label 名称，例如 HAS_POINT
  edge_table, // 承载关系实例的 Doris 边表；FK 型关系可复用 target 表
  source_label, // 关系源类型，例如 ICSProcess
  target_label, // 关系目标类型，例如 SCADAPoint
  source_key, // edge_table 中指向 source 实例 ID 的列
  target_key, // edge_table 中指向 target 实例 ID 的列
  outgoing_name, // source 侧导航伪列名，例如 ICSProcess.has_point
  incoming_name, // target 侧反向导航伪列名，例如 SCADAPoint.belongs_to_process
  outgoing_is_multi, // TRUE 表示 source.outgoing_name 类型为 MULTIROW<target_label>
  incoming_is_multi, // TRUE 表示 target.incoming_name 类型为 MULTIROW<source_label>
  cardinality, // 关系基数：one_to_one / one_to_many / many_to_one / many_to_many
  ai_context // 该谓语面向 LLM 的业务语义、别名、使用场景和反例
);
```

关系在语义层不暴露为让 LLM 手写的物理 join。

关系暴露为 `ROW<T>` / `MULTIROW<T>` 伪列。

映射规则：

```text
outgoing_is_multi = FALSE -> Source.outgoing_name ROW<Target>
outgoing_is_multi = TRUE  -> Source.outgoing_name MULTIROW<Target>
incoming_is_multi = FALSE -> Target.incoming_name ROW<Source>
incoming_is_multi = TRUE  -> Target.incoming_name MULTIROW<Source>
```

使用规则：

```text
ROW<T> 支持直接访问字段。
MULTIROW<T> 必须通过 JOIN / UNNEST / FLATTEN 展开。
```

### ABOX 数据层

ABOX DATA 存业务实例。

ABOX DATA 保持 Doris 表模型。语义层负责将 Doris 表投影为 TBOX 中的node\_type、relationship。

## 逻辑层

![image.png](https://alidocs.oss-cn-zhangjiakou.aliyuncs.com/res/pLdn55XRRzpN5no8/img/c7601735-569d-4dae-bbc0-e2c9a1e17c2c.png)

LOGIC 层覆盖以下能力：

*   业务规则定义。
    
*   指标定义。
    
*   DATA 属性和关系依赖声明。
    
*   SQL、UDF、Python、NLP 与外部服务实现方式声明。
    
*   外部大数据平台任务、模型或物化链路绑定。
    
*   ABOX 推理事实写回。
    
*   面向用户和 LLM 的结果解释。
    

在存在外部大数据平台时，LOGIC 层登记规则定义、依赖、外部任务绑定和结果位置。

任务的周期触发、资源调度、失败重试、运行日志和物化写回由外部平台负责。

LOGIC 层必须区分：

```text
TBOX Logic:
  规则定义。

ABOX Derived Facts:
  规则在具体实例上运行后的结果、证据和历史。
```

ABOX 侧术语采用：

```text
ABOX 推理事实
ABOX Derived Facts
ABOX Inferred Facts
```

### TBOX 逻辑层

TBOX LOGIC 存规则定义。

典型对象包括：

*   Semantic Property。
    
*   Semantic Measure。
    
*   State Machine。
    
*   Alert Rule。
    
*   Explanation Template。
    
*   External Job Binding。
    

实现方式支持：

```text
sql:
  GoogleSQL expression / query。

udf:
  Doris UDF 或 GoogleSQL UDF。

python:
  外部大数据平台中的 Python runtime，适合复杂算法。

nlp:
  NLP / LLM 辅助判断，适合非确定性文本理解。

external_service:
  调用外部大数据平台已有算法服务、规则引擎或预测服务。
```

工程原则：

*   Action 前置判断优先引用 deterministic SQL 或 UDF 产出的事实。
    
*   NLP / LLM 用于解释和辅助分类，不作为高风险 Action 前置条件的唯一依据。
    
*   Python / external service 的版本、输入、输出和运行状态由外部平台记录，本体层保存引用。
    
*   物化结果写回 Doris 或可查询数据源，并保留 evidence 引用。
    

#### Logic 定义表

```sql
ontology_logic (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  logic_name, // 规则名称，例如 ICSProcess.security_state
  target_type, // 作用目标类型，例如 ICSProcess
  target_property, // 输出属性，例如 security_state
  logic_kind, // 类型：semantic_property / measure / constraint / state_machine / alert_rule
  implementation_type, // 实现方式：sql / udf / python / nlp / external_service
  expression_sql, // SQL 表达式或查询；非 SQL 实现可为空
  udf_name, // UDF 名称；非 UDF 实现可为空
  python_entrypoint, // 外部平台 Python 入口引用；非 Python 实现可为空
  service_endpoint, // 外部平台服务 endpoint 引用；非 external_service 可为空
  deterministic, // 是否确定性；影响是否可用于 precondition
  execution_mode_hint, // 执行模式提示：on_read / materialized / scheduled / event_driven
  external_binding_name, // 外部执行绑定名称，例如 hai_security_state_job
  output_type, // 输出类型，例如 STRING / DOUBLE / BOOL / JSON
  ai_context // 面向 LLM 的规则含义、适用场景和反例
);
```

#### Logic 依赖表

```sql
ontology_logic_dependency (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  logic_name, // 规则名称
  dependency_kind, // 依赖类型：property / relation / measure / table / service
  dependency_name, // 依赖名称，例如 Equipment.active_alarm_count
  dependency_path, // 跨关系路径，例如 Equipment.has_sensor.latest_value
  required, // 是否必需
  description // 依赖说明
);
```

用途：

*   帮助编译器判断需要读取哪些 ABOX 数据。
    
*   帮助外部大数据平台理解哪些数据变化会触发重新计算。
    
*   帮助 LLM 解释一个结果来自哪些输入。
    

#### Logic 外部执行绑定表

```sql
ontology_logic_execution_binding (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  logic_name, // 规则名称
  platform_name, // 外部大数据平台名称，例如 FoundryLikePlatform
  platform_job_ref, // 外部平台任务 ID / pipeline ID / function ID
  platform_output_ref, // 外部平台输出数据集或模型输出引用
  result_table, // 写回 Doris 的结果表，例如 semantic_fact
  execution_mode_hint, // 执行模式提示：on_read / materialized / scheduled / event_driven
  trigger_rule_ref, // 外部平台触发规则引用；本体层不负责调度
  enabled, // 该绑定是否可被本体层使用
  owner, // 负责人或服务账号
  observability_ref // 外部平台运行记录、日志或监控地址引用
);
```

这张表只保存外部执行绑定，不负责启动定时任务。

```text
本体层:
  保存 logic 定义、依赖、输出位置、解释模板和外部任务引用。

外部大数据平台:
  负责调度、执行、重试、资源管理、任务日志和物化产出。

Doris:
  负责存储外部平台写回的 derived facts 或可查询结果表。
```

####  Logic 解释模板表

```sql
ontology_logic_explanation (
  domain_name, // 所属本体图名称
  version, // 所属 TBOX 版本
  logic_name, // 规则名称
  language, // 解释语言，例如 zh-CN / en-US
  template_text, // 解释模板
  evidence_schema_json, // evidence_json 的结构说明
  ai_context // 面向 LLM 的解释边界和禁止编造提示
);
```

解释模板示例：

```text
{process_id} 当前为 {security_state}，
因为攻击窗口数为 {attack_window_count_5m}，
受影响点位包括 {affected_points}，
命中了规则 {matched_rule}。
```

### ABOX 逻辑层

ABOX LOGIC 侧存储规则执行结果，统一称为 ABOX Derived Facts。

它包括：

*   当前推理事实。
    
*   历史推理事实。
    
*   规则运行记录。
    
*   证据 JSON。
    
*   解释文本。
    
*   计算时间。
    
*   规则版本。
    

示例：

```text
TBOX Logic:
  ICSProcess.security_state 计算规则。

ABOX Derived Fact:
  BoilerProcess(P1).security_state = AttackObserved。

Evidence:
  attack_label = true
  affected_points = ["P1_B2004"]
  
```

#### 推理事实表

```sql
semantic_fact (
  domain_name, // 所属本体图名称
  version, // 使用的 TBOX 版本
  object_type, // 对象类型，例如 Equipment
  object_id, // 对象 ID，例如 P1
  property_name, // 推理属性，例如 security_state
  value_type, // 值类型，例如 STRING
  value_string, // 字符串值
  value_number, // 数值
  value_bool, // 布尔值
  value_json, // 复杂值
  computed_by_logic, // 生成该事实的 logic_name
  computed_at, // 计算时间
  valid_from, // 事实生效开始时间
  valid_to, // 事实生效结束时间；当前事实可为空
  evidence_json, // 证据，例如报警数、温度、命中的规则
  provenance_json // 数据来源、任务 ID、输入版本等
);
```

#### Logic 运行记录表

```sql
semantic_logic_run (
  run_id, // 运行 ID
  domain_name, // 所属本体图名称
  version, // 使用的 TBOX 版本
  logic_name, // 规则名称
  execution_mode_hint, // 执行模式提示
  external_platform, // 外部大数据平台名称
  external_job_ref, // 外部平台任务引用
  started_at, // 开始时间
  finished_at, // 结束时间
  status, // 状态：running / success / failed / skipped
  input_count, // 输入对象数量
  output_count, // 输出事实数量
  error_message, // 失败信息
  metrics_json // 运行指标，例如耗时、扫描行数
);
```

## 动作层

ACTION 层覆盖以下能力：

*   LLM Agent 可调用业务动作声明。
    
*   Action 目标对象类型声明。
    
*   Action 输入输出 schema 声明。
    
*   Action 前置条件声明。
    
*   动作是否需要 dry-run。
    
*   外部大数据平台 action / workflow / function 绑定。
    
*   动作调用和执行结果记录。
    
*   外部动作结果回写。
    

ACTION 定义 LLM Agent 可调用的 tool / skill contract，包括目标对象、输入输出 schema、前置条件、dry-run 策略和外部动作绑定。

### TBOX 动作层

TBOX ACTION 定义动作能力。

存储内容包括：

*   Action 名称。
    
*   Tool 名称。
    
*   目标对象类型。
    
*   输入 schema。
    
*   输出 schema。
    
*   前置条件。
    
*   外部动作绑定。
    
*   调用模式。
    
*   LLM 使用说明。
    
    典型动作定义：
    
    ```text
    CreateIncidentTicket
      target_type = AttackWindow
      tool_name = create_incident_ticket
      external_platform = FoundryLikePlatform
      external_action_ref = security_workflow.create_incident_ticket
      requires_dry_run = true
    ```
    
    #### Action 定义表
    
    ```sql
    ontology_action (
      domain_name, // 所属本体图名称
      version, // 所属 TBOX 版本
      action_name, // Action 名称，例如 CreateIncidentTicket
      tool_name, // 暴露给 LLM 的工具名，例如 create_incident_ticket
      target_type, // 目标对象类型，例如 AttackWindow / ICSProcess
      input_schema_json, // 输入 schema，例如 severity / description / affected_points
      output_schema_json, // 输出 schema，例如 incident_id / status
      precondition_sql, // 前置条件 SQL，例如 attack_window.status != 'Closed'
      precondition_logic, // 前置条件引用的 logic，例如 ICSProcess.security_state
      external_platform, // 外部大数据平台名称，例如 FoundryLikePlatform
      external_action_ref, // 外部 action / workflow / function 引用
      invocation_mode, // 调用模式：dry_run_only / submit_request / execute_external
      dry_run_required, // 是否必须先 dry-run
      ai_context // 面向 LLM 的调用时机、禁止调用场景和业务提示
    );
    ```
    
    #### Action 外部绑定表
    
    ```sql
    ontology_action_binding (
      domain_name, // 所属本体图名称
      version, // 所属 TBOX 版本
      action_name, // Action 名称
      platform_name, // 外部大数据平台名称
      platform_action_ref, // 外部 action / workflow / function ID
      dry_run_ref, // 外部 dry-run / preview endpoint 引用
      result_ref, // 外部执行结果或状态查询引用
      observability_ref, // 外部运行记录、日志或监控地址引用
      enabled // 该绑定是否可用
    );
    ```
    
    ### ABOX 动作层
    
    #### Action 实例表
    
    ```sql
    semantic_action_instance (
      action_id, // Action 实例 ID
      domain_name, // 所属本体图名称
      version, // 使用的 TBOX 版本
      action_name, // Action 名称
      tool_name, // Tool 名称
      target_type, // 目标对象类型
      target_object_id, // 目标对象 ID
      input_json, // 输入参数
      dry_run, // 是否 dry-run
      status, // 状态：pending / submitted / running / success / failed / cancelled
      precheck_result_json, // 前置条件检查结果
      external_request_ref, // 外部平台请求 ID
      external_result_json, // 外部平台执行结果
      requested_by, // 请求人
      requested_by_agent, // 请求 Agent
      created_at, // 创建时间
      updated_at // 更新时间
    );
    ```
